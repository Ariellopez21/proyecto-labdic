\section{Arquitectura general y panorama del proyecto}

La arquitectura global del sistema describe el recorrido que realizan los datos desde la base de datos hasta la interfaz de usuario. En primer lugar, la información se almacena en PostgreSQL y se modela mediante SQLAlchemy, que actúa como ORM para mapear las tablas a clases de Python. Sobre esta capa se apoya el framework Litestar, encargado de exponer los modelos a través de servicios API con operaciones CRUD para cada recurso del sistema. De esta manera, la lógica de negocio queda encapsulada en servicios bien definidos que centralizan la validación y el acceso a los datos.

A partir de estas APIs, el backend se comunica con el frontend. En la capa de presentación se utiliza Vue 3 con Composition API, donde los componentes consumen las APIs mediante interfaces tipadas y solicitudes HTTP que se ejecutan cada vez que la aplicación requiere leer o modificar información. Los datos recibidos se procesan en el cliente, se integran al estado global de la aplicación y finalmente se representan en la interfaz gráfica a través de componentes de UI que facilitan una lectura clara y consistente para las personas que utilizan el sistema. \section{Implementación del módulo de usuarios}\label{sec:modulo_usuario_backend}
Esta sección constituye el núcleo del hito, ya que describe cómo el diseño lógico definido en la base de datos se traduce en código ejecutable tanto en el backend como en el frontend. A partir del modelo Entidad–Relación y de las decisiones arquitectónicas descritas previamente, se implementa un conjunto de servicios que permiten crear (\texttt{POST}), leer (\texttt{GET}), actualizar (\texttt{PUT/PATCH}) y eliminar (\texttt{DELETE}) usuarios, gestionar sus roles y controlar el acceso al sistema. 

Las subsecciones siguientes detallan, en primer lugar, la selección de herramientas y librerías empleadas; luego, la estructura interna del backend (modelos, servicios y controladores); más adelante, la implementación del frontend y su integración con la API; y finalmente, las pruebas realizadas y los resultados obtenidos.

\subsection{Elección de herramientas y librerías}

Para la implementación del módulo de usuarios se optó por un conjunto de herramientas que permiten equilibrar rendimiento, claridad en el código y buenas prácticas de seguridad. La mayor parte de las justificaciones conceptuales ya se presentó en el marco teórico (sección~\ref{sec:marco_teorico})
del marco teórico; en esta subsección se enfatiza su rol concreto dentro del módulo.

En la capa de backend se utiliza \textbf{Litestar} como framework principal. Al tratarse de un framework ASGI ~\cite{plainenglish_litestar_fastapi} 
moderno orientado a la construcción de APIs, facilita la definición de rutas, controladores y esquemas de datos fuertemente tipados en Python. Esto permite describir de manera declarativa los endpoints del módulo de usuarios (por ejemplo, registro, inicio de sesión, actualización de datos y asignación de roles), aprovechando características como inyección de dependencias, manejo centralizado de excepciones y generación automática de documentación OpenAPI. De este modo, la lógica asociada al usuario se organiza en servicios reutilizables y fáciles de probar, que luego podrán servir como plantilla para los módulos de productos y préstamos.

En cuanto a la \textbf{gestión segura de contraseñas}, se emplea \texttt{argon2} como algoritmo de hashing a través de la librería \texttt{pwdlib}. Argon2 es un algoritmo moderno diseñado específicamente para almacenar contraseñas de forma segura~\cite{master_spring_ter_2025_argon2}
, ya que introduce un costo computacional y de memoria configurable que dificulta los ataques de fuerza bruta y de diccionario. La integración con \texttt{pwdlib} simplifica la creación y verificación de hashes, encapsulando detalles como la generación de \textit{salts}, la elección de parámetros de seguridad y la comparación constante en tiempo, reduciendo así el riesgo de errores de implementación que comprometan la seguridad del sistema.

Para la \textbf{autenticación y autorización} se utiliza \texttt{OAuth2} que proviene de litestar como \texttt{litestar.contrib.jwt}. Esta librería permite implementar un flujo basado en \emph{bearer tokens} que encapsulan la identidad del usuario autenticado y la información necesaria para autorizar solicitudes posteriores. De esta forma, el módulo de usuarios expone endpoints de inicio de sesión que generan tokens de acceso, y el resto de los servicios del sistema (tanto actuales como futuros) pueden verificar dichos tokens antes de procesar cualquier operación sensible. Esta aproximación desacopla la autenticación de la lógica de negocio específica de cada módulo y favorece la extensibilidad, por ejemplo, se podrían incorporar \texttt{refresh\_tokens}, roles más complejos o incluso integración con proveedores externos de identidad sin reescribir la base del sistema. 

Así, las herramientas mencionadas: Litestar, \texttt{pwdlib} con Argon2 y \texttt{OAuth2}; Integran un ecosistema coherente para el módulo de usuarios, donde permiten estructurar la API de forma clara, proteger correctamente las credenciales y establecer un mecanismo estándar de emisión y verificación de tokens que será compartido por los demás módulos del proyecto.

\subsection{Estructura jerárquica}\label{sec:estructura_jerarquica}

El sistema adopta una estructura jerárquica basada en \emph{routers} tanto en el backend como en el frontend. En el backend, con Litestar se utiliza directamente la clase \texttt{Router} de la librería principal \texttt{litestar}, lo que permite agrupar endpoints relacionados bajo prefijos comunes y aplicar configuraciones compartidas (como dependencias, middlewares o políticas de seguridad). En el frontend, se emplea \texttt{vue-router} mediante la función \texttt{createRouter} y la interfaz \texttt{RouteRecordRaw}, que actúa como estructura de datos para definir el árbol completo de rutas, sus componentes asociados y las opciones de navegación. Esta combinación facilita el manejo de rutas, el control de acceso y la organización modular de las distintas secciones del sistema.

En el frontend, el control de acceso se refuerza utilizando la información de autenticación y los roles del usuario. A través de servicios de autenticación se determina, antes de cada cambio de ruta, si el usuario posee los privilegios necesarios para acceder a una vista determinada. De este modo, se evita que usuarios sin permisos intenten llegar a páginas restringidas, incluso si manipulan manualmente la URL o realizan intentos de acceso forzado.

La estructura jerárquica de las rutas en el backend se diseña considerando la escalabilidad futura del sistema. Se contempla un \texttt{root} vacío para permitir que, si el sistema crece, pueda añadirse un prefijo común para todo el módulo de inventario, por ejemplo \texttt{/labdic\_inventory}. A partir de este prefijo se organizan los distintos recursos: \texttt{/labdic\_inventory/users} para el recurso de usuarios, \texttt{/labdic\_inventory/users/5} para acceder al usuario con identificador 5, o \texttt{/labdic\_inventory/users/me} para obtener la información del usuario autenticado. Esta convención uniforme facilita la comprensión y el versionado futuro de la API.

En el frontend, la estructura jerárquica se orienta hacia rutas descriptivas que favorezcan una navegación lógica y legible. Desde la vista principal, ubicada en el \texttt{root} del frontend, se accede a cada servicio mediante rutas como \texttt{/users}, \texttt{/users/me} o \texttt{/users/5}, manteniendo la coherencia con la API del backend pero adaptada a las necesidades de la interfaz. Para operaciones específicas se definen rutas adicionales, por ejemplo \texttt{/users/create} para la creación de usuarios o \texttt{/users/update/5} para la edición de un usuario concreto. El control de acceso se encarga de que un usuario estándar no pueda acceder a rutas como \texttt{/users/create} o \texttt{/users/update/:id}, aun cuando conozca la URL.

Finalmente, se proyecta un conjunto de rutas específicas para administración, agrupadas bajo un prefijo común como \texttt{/admin}. Allí se concentrarán las vistas y operaciones exclusivas de perfiles con mayores privilegios, tales como la asignación de roles, la gestión de productos o el ajuste de stock. Por ejemplo, la ruta \texttt{/admin/roles} permitirá a un administrador visualizar todos los roles, modificarlos, crear nuevos o eliminar aquellos que ya no sean necesarios. Esta separación clara entre rutas generales y rutas administrativas contribuye tanto a la seguridad como a la mantenibilidad del sistema.


\subsection{Modelado de datos, servicios y operaciones}\label{sec:modelado_datos}

El punto de partida para la implementación del módulo de usuarios es la estructura del modelo \texttt{users}, construida y explicada en el hito~2. Dicho modelo forma parte del diagrama Entidad–Relación general del sistema, donde se aprecian las relaciones entre usuarios y las demás entidades del laboratorio. En la Figura~\ref{fig:diagrama_modelo_entidad_relacion} puede observarse la posición central de \texttt{users} dentro del modelo de datos y cómo se vincula con tablas como \texttt{roles}, \texttt{loan\_request} y \texttt{device\_status\_log}.

En términos de relaciones, la entidad \texttt{users} se conecta con otras tablas de la siguiente manera:

\begin{itemize}
	\item \textbf{\texttt{roles}}: La relación entre \texttt{users} y \texttt{roles} es de muchos a muchos (M:N), lo que da lugar a una tabla intermedia \texttt{user\_roles}. Esta estructura permite que un usuario tenga más de un rol y, simultáneamente, un mismo rol pueda asignarse a múltiples usuarios. De esta manera se cubren escenarios como:
	\begin{itemize}
		\item Un usuario con rol de \emph{profesor} que además posee rol de \emph{admin}.
		\item Un estudiante que también cumple funciones de \emph{ayudante}.
	\end{itemize}
	En el estado actual del sistema, los roles que generan una distinción funcional clara son \texttt{admin} y \texttt{user}, pues determinan el comportamiento de superusuario y las capacidades de administración. No obstante, gracias a este diseño y al manejo de control de acceso que se abordará en la implementación del frontend, la incorporación futura de nuevos roles (por ejemplo, \emph{profesor} con capacidad de gestionar solicitudes de salas para clases extraordinarias, en un eventual módulo de reservas) será modular y segura, sin necesidad de modificar la estructura base.
	
	\item \textbf{\texttt{loan\_request}}: La relación entre \texttt{users} y \texttt{loan\_request} es de uno a muchos (1:M), ya que cada usuario puede generar múltiples solicitudes de préstamo. Cada registro en \texttt{loan\_request} almacena información relevante para que un usuario administrador pueda evaluar y aceptar la solicitud: detalles del equipo solicitado, estado de la solicitud, así como marcas temporales que indican la fecha de creación, la fecha de entrega del equipo, la fecha estimada de devolución y la fecha de retorno real. Este diseño facilita el seguimiento de cada préstamo a lo largo de todo su ciclo de vida.
	
	\item \textbf{\texttt{device\_status\_log}}: También se establece una relación de uno a muchos (1:M) entre \texttt{users} y \texttt{device\_status\_log}. En este caso se utiliza el \texttt{id} de usuario para identificar quién tiene en posesión un dispositivo específico en un momento dado. Por ejemplo, si se presta un \emph{notebook}, la tabla de estados registra que dicho equipo se encuentra asociado a un usuario concreto, reflejando la situación actual del inventario. Esto permite que, al revisar la sección de \emph{notebooks}, un administrador pueda ver de forma clara qué dispositivos están prestados, a quién y bajo qué condiciones.
\end{itemize}

En el backend, los \textbf{servicios} se estructuran siguiendo el principio de que, por cada tabla principal, exista un servicio dedicado, con la excepción del servicio \texttt{auth}, que agrupa la lógica transversal de autenticación. En el contexto del módulo de usuarios, se definen servicios específicos para \texttt{users} y \texttt{roles}, cada uno de ellos compuesto por:

\begin{itemize}
	\item Un \textbf{DTO} (Data Transfer Object) que define la forma en que los datos viajan entre el backend y el frontend, controlando qué campos se exponen y en qué contexto.
	\item Un \textbf{repositorio} que encapsula las operaciones de acceso a datos sobre la tabla correspondiente.
\end{itemize}

Para construir estos componentes se utilizan las siguientes librerías asociadas a Litestar y Advanced Alchemy:

\begin{itemize}
	\item \texttt{advanced\_alchemy.extensions.litestar}: utilidades para integrar repositorios y DTOs con Litestar.
	\item \texttt{litestar.plugins.sqlalchemy}: soporte para trabajar con SQLAlchemy y generar DTOs a partir de los modelos.
	\item \texttt{advanced\_alchemy.repository}: base para la creación de repositorios reutilizables, con operaciones CRUD ya implementadas.
	\item \texttt{litestar.dto}: herramientas para definir y personalizar DTOs usados en los endpoints.
\end{itemize}

El uso de \textbf{repositorios} tiene como objetivo principal aprovechar métodos genéricos provistos por estas librerías para reducir la duplicación de código SQL manual y centrar el esfuerzo de desarrollo en la lógica de negocio. De esta forma, las operaciones CRUD básicas (crear, obtener, actualizar y eliminar) se apoyan en repositorios compartidos y consistentes, y la lógica más compleja (validaciones específicas, reglas de negocio, comprobación de roles, etc.) se implementa en capas superiores. Además, los repositorios pueden reutilizarse entre servicios; por ejemplo, el servicio de usuarios puede utilizar el repositorio de \texttt{roles} para verificar rápidamente qué roles posee un usuario y aplicar restricciones de acceso sin reescribir consultas.

Los \textbf{DTOs}, por su parte, permiten controlar la visibilidad de los campos de cada modelo según el contexto. Por ejemplo, al exponer información de un usuario en una ruta pública o semipública, el DTO puede ocultar el campo de contraseña y otros datos sensibles sin necesidad de modificar la tabla subyacente. Esto mejora la seguridad y la claridad del contrato que el backend ofrece al frontend.

Finalmente, la capa de \textbf{controladores} se estructura con un controlador por cada servicio principal. Cada controlador:

\begin{itemize}
	\item Define las rutas y métodos HTTP (idealmente abarcando al menos \texttt{GET}, \texttt{POST}, \texttt{PUT}/\texttt{PATCH} y \texttt{DELETE}) que componen la API del recurso.
	\item Invoca a los repositorios y DTOs correspondientes, delegando en ellos la lógica de acceso a datos y la transformación de estructuras.
	\item Incluye documentación y descripciones claras de cada endpoint, de manera que el comportamiento del servicio sea comprensible con una lectura rápida del archivo.
\end{itemize}

Bajo esta organización, los controladores se mantienen livianos y legibles, mientras que la lógica pesada se concentra en los repositorios y en la definición de los DTOs. Esto no sólo mejora la mantenibilidad del módulo de usuarios, sino que además ofrece un patrón reutilizable para los futuros servicios asociados a los módulos de productos y préstamos, favoreciendo la homogeneidad de la arquitectura en todo el sistema.



\subsection{Servicios y operaciones implementados}

En el módulo de usuarios se implementaron tres servicios principales: \texttt{users}, \texttt{roles} y \texttt{auth}. Los dos primeros comparten una estructura similar, basada en la exposición de operaciones CRUD a través de métodos HTTP estándar, mientras que el servicio de \texttt{auth} concentra la lógica de autenticación y emisión de tokens.

Tanto para \texttt{users} como para \texttt{roles}, las operaciones se apoyan en los repositorios descritos previamente, aprovechando los métodos genéricos que estos ofrecen:

\begin{itemize}
	\item \textbf{GET}: se utilizan los métodos \texttt{list} y \texttt{get\_one}. El primero permite obtener un conjunto de registros (equivalente a un \texttt{SELECT *} sobre la tabla), eventualmente con filtros o paginación, mientras que el segundo está orientado a recuperar una fila específica mediante su \texttt{id}. Estas operaciones se exponen en endpoints como \texttt{GET /users} o \texttt{GET /roles/:id}.
	
	\item \textbf{POST}: para crear nuevos registros se utiliza el método \texttt{add}, que inserta una nueva fila en la tabla asociada al modelo. En el caso de \texttt{users}, esto implica recibir un DTO con los datos del usuario, aplicar las validaciones correspondientes (incluido el hashing de la contraseña) y persistir el resultado en la base de datos.
	
	\item \textbf{PATCH}: para la actualización de datos se opta por el uso de \texttt{PATCH} en lugar de \texttt{PUT}. La razón principal es que \texttt{PATCH} permite actualizaciones parciales, es decir, modificar únicamente algunos campos sin necesidad de reescribir la fila completa. Para ello se emplea el método \texttt{get\_and\_update}, que primero obtiene la instancia del modelo y luego aplica los cambios indicados en el cuerpo de la petición. Esto resulta especialmente útil cuando se desea, por ejemplo, cambiar sólo el apellido o el correo electrónico de un usuario, manteniendo intactos otros campos.
	
	\item \textbf{DELETE}: finalmente, para eliminar registros se utiliza el método \texttt{delete}, que remueve la fila correspondiente a un \texttt{id} determinado. De esta forma se implementan operaciones como \texttt{DELETE /users/:id} o \texttt{DELETE /roles/:id}.
\end{itemize}

Todos estos métodos se configuran tanto para \texttt{users} como para \texttt{roles}, reutilizando la infraestructura de repositorios definida con Litestar y SQLAlchemy. Esto asegura una interfaz uniforme para las operaciones CRUD y reduce la cantidad de lógica específica que debe escribirse en cada controlador.

En el caso particular de \texttt{users}, además de los endpoints CRUD estándar, se implementa una operación GET adicional, \texttt{get\_my\_user}, que permite obtener la información del usuario actualmente autenticado. Para ello, la operación recibe el objeto \texttt{request}, desde el cual se extrae el usuario activo asociado al token de acceso enviado en la cabecera de la petición. De esta manera, un cliente puede invocar, por ejemplo, \texttt{GET /users/me} para recuperar sus propios datos sin necesidad de conocer su \texttt{id} numérico. El detalle de cómo se gestionan los tokens y la autenticación subyacente se desarrolla en la sección~\ref{sec:autenticacion_seguridad}.

De forma más concreta, el controlador de usuarios definido en \texttt{controllers.py} expone las siguientes rutas y operaciones:

% Fuente: definición de controladores de usuarios. :contentReference[oaicite:0]{index=0}
\begin{itemize}
	\item \textbf{\texttt{GET /users/me}} (\texttt{get\_my\_user}): recibe un \texttt{Request[User, Token, Any]} e interpreta el usuario autenticado a partir del token JWT. No requiere parámetros adicionales ni cuerpo en la petición; retorna un objeto \texttt{User} serializado mediante \texttt{UserReadDTO}. La validación principal consiste en la presencia de un token válido y no expirado.
	
	\item \textbf{\texttt{GET /users/}} (\texttt{list}): no recibe parámetros de ruta ni cuerpo; únicamente se inyecta el \texttt{UserRepository} mediante dependencias. Retorna una secuencia de \texttt{User} representados con \texttt{UserReadDTO}. Su finalidad es listar todos los usuarios registrados en el sistema.
	
	\item \textbf{\texttt{GET /users/\{user\_id:int\}}} (\texttt{fetch}): define un parámetro de ruta \texttt{user\_id} de tipo entero. El controlador utiliza \texttt{users\_repo.get\_one(id=user\_id)} para recuperar el registro correspondiente. Si el usuario no existe, se lanza \texttt{NotFoundError}, que es manejado por \texttt{not\_found\_error\_handler} devolviendo una respuesta \texttt{404} con un mensaje JSON descriptivo. El esquema de salida vuelve a ser \texttt{UserReadDTO}, ocultando campos sensibles.
	
	\item \textbf{\texttt{POST /users/}} (\texttt{create}): protegido por la guarda \texttt{admin\_user\_guard}, por lo que sólo usuarios con \texttt{is\_admin = True} pueden invocarlo. La operación recibe un cuerpo JSON validado mediante \texttt{UserCreateDTO}, que se transforma en una instancia de \texttt{User}. Además, se inyectan \texttt{UserRepository} y \texttt{RoleRepository} como dependencias. Internamente se llama a
	
	\texttt{users\_repo.add\_with\_existing\_roles}, lo que permite asociar al nuevo usuario roles ya existentes en la base de datos. Las reglas de validación (campos obligatorios, formatos, restricciones de tipo) se declaran en \texttt{UserCreateDTO}; el controlador se limita a orquestar la operación y devolver el usuario creado a través de \texttt{UserReadDTO}.
	
	\item \textbf{\texttt{PATCH /users/\{user\_id:int\}}} (\texttt{update}): también protegido por 
	
	\texttt{admin\_user\_guard}. El parámetro de ruta \texttt{user\_id} identifica el usuario a modificar, mientras que el cuerpo de la petición se valida con 
	
	\texttt{UserUpdateDTO}, que permite actualizaciones parciales mediante 
	
	\texttt{DTOData[User]}. La operación delega en 
	
	\texttt{users\_repo.update\_with\_existing\_roles}, actualizando tanto los campos del usuario como la asociación de roles, si corresponde. Nuevamente, las reglas de validación (qué campos son opcionales, rangos válidos, tipos, etc.) residen en el DTO y en el modelo de dominio.
	
	\item \textbf{\texttt{DELETE /users/\{user\_id:int\}}} (\texttt{delete}): restringido a administradores mediante \texttt{admin\_user\_guard}. Recibe únicamente el parámetro de ruta \texttt{user\_id} y utiliza \texttt{users\_repo.delete} para eliminar el registro correspondiente. El método retorna \texttt{None}, por lo que la respuesta HTTP esperada es un estado de éxito sin contenido (generalmente \texttt{204 No Content}). Este endpoint no expone un esquema de salida, y su validación se centra en que el usuario tenga privilegios de administrador y que el \texttt{user\_id} apunte a un registro existente.
\end{itemize}

Esta descripción detallada por operación permite comprender no sólo qué acciones se exponen a través de la API, sino también qué parámetros aceptan, qué esquemas de entrada y salida utilizan y qué reglas de validación se aplican. Además, establece un patrón claro que podrá replicarse al diseñar los controladores para los módulos de productos y préstamos, manteniendo consistencia en el diseño de la API del sistema.

\subsection{Modularidad y escalabilidad}

La modularidad y la escalabilidad del sistema se apoyan fuertemente en el uso combinado de SQLAlchemy y Alembic. Por un lado, SQLAlchemy proporciona una capa de abstracción sobre la base de datos que permite definir el esquema mediante modelos declarativos en Python, mientras que Alembic se encarga de gestionar la evolución controlada de ese esquema a través de migraciones versionadas. Esta combinación facilita tanto la creación inicial de las tablas como su modificación progresiva a medida que el sistema crece y se incorporan nuevos requerimientos.

En primer lugar, \textbf{SQLAlchemy} permite describir cada entidad del dominio (por ejemplo, \texttt{users}, \texttt{roles}, \texttt{loan\_request} o \texttt{device\_status\_log}) como una clase de Python, donde se declaran sus columnas, tipos de datos, restricciones y relaciones. Esto aporta varias ventajas:

\begin{itemize}
	\item El esquema se mantiene cercano al código, lo que hace más sencillo razonar sobre las entidades y sus relaciones al leer el código, sin depender exclusivamente de scripts SQL sueltos.
	\item La definición de relaciones (\texttt{ForeignKey}, \texttt{relationship}, tablas intermedias para relaciones M:N, etc.) queda centralizada y reutilizable, de modo que cualquier servicio que consuma esos modelos lo hace con una API homogénea.
	\item El acceso a datos se realiza a través de un ORM, lo que simplifica operaciones complejas (joins, filtros, paginación) y evita repetir consultas SQL manuales, facilitando la creación de repositorios genéricos como los utilizados para usuarios y roles.
\end{itemize}

Sobre esta base, \textbf{Alembic} actúa como herramienta de migraciones que registra y aplica los cambios en el esquema de la base de datos de forma incremental. Cada vez que se modifica un modelo en SQLAlchemy (por ejemplo, añadiendo una columna, ajustando una relación o creando una nueva tabla para un módulo futuro), se genera una migración que describe ese cambio en términos de \emph{upgrade} y \emph{downgrade}. Esto tiene implicancias directas en la escalabilidad del proyecto:

\begin{itemize}
	\item Se puede \emph{evolucionar} el esquema sin perder el historial, manteniendo un conjunto ordenado de versiones que permiten llevar la base de datos hacia adelante o hacia atrás según sea necesario.
	\item Las nuevas funcionalidades (como la incorporación de un módulo de salas de clases o la ampliación de los datos de préstamos) se pueden introducir mediante migraciones puntuales, evitando rediseños globales o cambios manuales riesgosos.
	\item Es posible sincronizar distintos entornos (desarrollo, pruebas, producción) aplicando las mismas migraciones en orden, lo que garantiza coherencia entre las instancias de la base de datos y minimiza errores de despliegue.
\end{itemize}

Sobre el módulo de usuarios, esto permite que la estructura actual, centrada en \texttt{users} y \texttt{roles}, sirva de base para futuras ampliaciones. Si más adelante se requieren nuevos campos (por ejemplo, datos adicionales de perfil) o nuevas relaciones (como asociaciones con módulos externos), bastará con ajustar los modelos de SQLAlchemy y generar las migraciones correspondientes en Alembic. De este modo, la arquitectura de datos sigue siendo modular y extensible, acompañando el crecimiento del sistema sin sacrificar estabilidad ni claridad en la definición del esquema.

\subsection{Autenticación y seguridad}\label{sec:autenticacion_seguridad}

La implementación de la autenticación y la gestión de permisos en el sistema se articula a través de tres elementos fundamentales: el controlador de autenticación, las guards y controladores de usuarios y los servicios de seguridad encargados de gestionar tokens y contraseñas. 

\subsubsection*{Controlador de autenticación.}
El controlador de la autenticación define una clase \texttt{AuthController} con la ruta base \texttt{/auth}. El endpoint \texttt{POST /login} recibe datos de usuario y contraseña a través de un DTO (\texttt{UserLoginDTO}) y una dependencia de \texttt{UserRepository}. La lógica de este método \texttt{login} consiste en buscar un usuario por nombre en la base de datos (\texttt{users\_repo.get\_one\_or\_none}) y, si existe, verificar que la contraseña proporcionada coincide con la almacenada mediante \texttt{users\_repo.check\_password}. En caso de credenciales inválidas, se genera una excepción \texttt{HTTPException} con código 401; si son correctas, se invoca a \texttt{oauth2\_auth.login}, que emite un token JWT (Json Web Token) para el usuario autenticado. Este controlador es, por tanto, la puerta de entrada al sistema: encapsula la lógica de inicio de sesión, delega la verificación de credenciales al servicio de usuarios y se apoya en el objeto de seguridad global para generar el token. De este modo, el servicio de usuarios sigue siendo responsable de la persistencia y validación de contraseñas, mientras que el controlador de autenticación se limita a orquestar la emisión de tokens y a centralizar los mensajes de error en caso de autenticación fallida.

\subsubsection*{Guards y controladores de usuarios.}
El controlador de \texttt{users} define un controlador \texttt{UserController} para la ruta base \texttt{/users}. Una de sus características más importantes es la función \texttt{admin\_user\_guard}, que se utiliza como guard de acceso en aquellas operaciones que requieren privilegios de administrador (creación, actualización y eliminación de usuarios). Esta función recibe una conexión \texttt{ASGIConnection} y un \texttt{BaseRouteHandler} y verifica la propiedad que corresponde a la identificación de un usuario como administrador; sí la condición no se cumple, lanza una \texttt{HTTPException} con código 403. La elección de \texttt{ASGIConnection} se debe a que Litestar es un framework basado en ASGI, lo que significa que cada solicitud HTTP se representa como una conexión asíncrona que contiene información de contexto (usuario autenticado, cabeceras, etc.). Al emplear \texttt{ASGIConnection} en la guard, se obtiene acceso al usuario ya resuelto por el middleware de autenticación, de modo que la comprobación de roles se realiza de manera eficiente y coherente en todas las rutas protegidas. La clase \texttt{BaseRouteHandler}, aunque no se usa directamente en el código, forma parte de la firma de la guard y permite que Litestar inyecte información sobre la ruta y el manejador, habilitando posibles extensiones futuras como mensajes de error.

\subsubsection*{Gestión de seguridad y tokens.}
El archivo dedicado a la administración de la seguridad del backend nombrado como \texttt{security.py} centraliza la configuración de autenticación mediante el patrón \texttt{OAuth2PasswordBearerAuth}. Se define una función \texttt{retrieve\_user\_handler} que, a partir del \texttt{token} recibido en la solicitud, abre una sesión de base de datos, construye un \texttt{UserRepository} y recupera el usuario correspondiente mediante \texttt{get\_my\_user}. Si el usuario no se encuentra, se lanza una \texttt{NotFoundException}. Este handler se pasa al \texttt{Bearer} de autenticación junto con el token secreto de registro, la ruta de login y una lista de rutas excluidas de autenticación. El resultado es un objeto \texttt{oauth2\_auth} que se inyecta como dependencia en el controlador de autenticación y en cualquier otra parte del sistema que requiera validar y extraer usuarios a partir de un JWT. Gracias a este mecanismo, cada solicitud autenticada dispone de un contexto \texttt{request.user} completo, que a su vez es accesible desde \texttt{ASGIConnection} en las guards.

\subsubsection*{Hashing de contraseñas y relación con los roles.}
Finalmente, en el manejo del repositorio del modelo \texttt{users}, se vincula el almacenamiento seguro de contraseñas con la asignación de roles y la autenticación. La clase \texttt{UserRepository} extiende \texttt{SQLAlchemySyncRepository} y define, entre otros, los métodos \texttt{add\_with\_existing\_roles} y \texttt{check\_password}. El primero recibe un \texttt{User} y un \texttt{RoleRepository}; antes de insertar el usuario, aplica el método \texttt{password\_hasher.hash} a su contraseña mediante \texttt{pwdlib}, garantizando que se almacene de forma segura con un algoritmo moderno de hashing (por defecto, Argon2). A continuación, reemplaza la lista de roles del usuario por una lista de objetos persistentes recuperados a partir de sus \texttt{id} mediante \texttt{roles\_repo.list}. De este modo se asegura que el usuario quede vinculado únicamente a roles válidos existentes en la base de datos. El método \texttt{check\_password}, por su parte, recupera al usuario por nombre de usuario y verifica la contraseña proporcionada mediante \texttt{password\_hasher.verify}. La combinación de estas funciones permite que el controlador de autenticación delegue toda la lógica de hashing y comprobación al repositorio, manteniendo una única fuente de verdad para la seguridad de las credenciales.

Un aspecto importante de esta implementación es que la asignación de roles se maneja de forma robusta y opcional: si un usuario no trae roles asociados, el método \texttt{add\_with\_existing\_roles} simplemente construye una lista vacía, lo que evita errores en tiempo de ejecución y permite que el módulo de usuarios funcione incluso si la capa de roles se desactiva temporalmente. Esta arquitectura modular hace posible introducir o eliminar roles en el futuro sin modificar la estructura básica del modelo \texttt{User}; las validaciones de rol y las restricciones de acceso se aplican en las guards del controlador (\texttt{admin\_user\_guard}), mientras que el repositorio sólo se encarga de enlazar instancias persistentes de roles existentes.

\section{Implementación del frontend para el módulo de usuarios}

\subsection{Elección de herramientas y librerías}

El desarrollo del frontend se basa en \textbf{Vue 3} como framework principal. Esta elección responde a la experiencia acumulada durante la carrera y a la robustez del ecosistema de Vue, que se caracteriza por una comunidad activa y una arquitectura altamente modular. Vue 3 permite organizar la lógica de componentes mediante la Composition API, que ayuda a separar y reutilizar funcionalidades de forma más clara que el modelo tradicional, lo cual se ajusta perfectamente a la estructura modular del proyecto. 

La gestión de estados globales se resuelve con \textbf{Pinia}, el gestor oficial para Vue 3. Pinia ofrece un API sencilla y reactiva, con soporte completo para TypeScript, lo que permite modelar las stores de usuarios, roles y autenticación con tipado fuerte y facilitar la depuración. 

Para la interfaz de usuario se eligió la biblioteca de componentes \textbf{PrimeVue}, que dispone de un catálogo amplio de elementos accesibles y fácilmente personalizables. Su combinación con \textbf{TailwindCSS} permite componer diseños modernos mediante clases utilitarias sin definir estilos CSS personalizados, facilitando la coherencia visual de la aplicación. Estas dos herramientas se complementan: PrimeVue proporciona componentes listos para usarse (formularios, diálogos, tablas), mientras que TailwindCSS aporta control granular sobre el diseño y responsive design. 

El proyecto se compila y empaqueta con \textbf{Vite}, que ofrece un servidor de desarrollo extremadamente rápido gracias a la importación de módulos ES y un sistema de \emph{hot module replacement}~\cite{vite_why_vite_dev_server}. Vite emplea Rollup para construir los artefactos finales, permitiendo aprovechar las optimizaciones de empaquetado sin perder la agilidad en el desarrollo~\cite{vite_why_vite_rollup}. Además, Vite simplifica la configuración al detectar la estructura de un proyecto Vue y facilitar la importación automática de componentes y dependencias.~\cite{vite_build_rollup_options}~\cite{devto_vite_vs_webpack} 

La comunicación con el backend se realiza mediante la Composition API, que encapsulan las funciones de consumo de APIs. Esto, armoniza con la definición de DTOs en Litestar: cada interfaz en TypeScript dentro de \texttt{Type Interface} refleja las estructuras validadas por el backend como se apreció en la sección~\ref{sec:modelado_datos}, reduciendo el riesgo de desajustes entre ambas capas.

\subsection{Arquitectura del frontend y estructura de carpetas}

La estructura de carpetas del frontend, basada en Vue 3 y Vite, se organiza para separar claramente las responsabilidades y facilitar la escalabilidad. El árbol de directorios general (ignorando \texttt{.vscode/} y \texttt{node\_modules/}) es el siguiente:

\begin{lstlisting}[language=bash, numbers=none]
	app-frontend/
	|-- public/
	|-- src/
	|   |-- api/
	|   |-- assets/
	|   |-- components/
	|   |   |-- base/
	|   |   |-- forms/
	|   |   |-- dialogs/
	|   |   |-- tables/
	|   |-- interfaces/
	|   |-- plugins/
	|   |-- router/
	|   |-- stores/
	|   |-- views/
	|   |-- App.vue
	|   `-- main.ts
	|-- .editorconfig
	|-- .env
	|-- auto-imports.d.ts
	|-- components.d.ts
	|-- env.d.ts
	|-- index.html
	`-- otros_archivos
\end{lstlisting}

En esta estructura, cada directorio tiene una función específica. El directorio \texttt{api/} alberga los servicios de consumo de la API, centralizando las llamadas HTTP hacia el backend. Estos servicios se implementan como \emph{composables} reutilizables; cada archivo encapsula funciones para obtener, crear, actualizar o eliminar entidades (como \texttt{users} o \texttt{roles}) y maneja, además, estados de carga y errores asociados a cada petición. Por su parte, \texttt{interfaces/} contiene los tipos e \texttt{interface} de TypeScript que definen la estructura de los datos intercambiados con el servidor. Estas definiciones reflejan los DTOs expuestos por Litestar, asegurando la correspondencia entre los datos recibidos y los esperados por los componentes y las stores. El directorio \texttt{stores/} reúne las stores de Pinia, donde cada una gestiona un ámbito concreto de estado; por ejemplo, \texttt{useUserStore} concentra la información del usuario autenticado. Las acciones de estas stores invocan los servicios definidos en \texttt{api/} y actualizan el estado global según corresponda.

El directorio \texttt{router/} contiene la configuración de Vue Router. En él se definen las rutas de la aplicación, los componentes asociados a cada ruta y las guardas de navegación necesarias para controlar el acceso. Estas guardas utilizan los estados de autenticación y los roles definidos en las stores para decidir si un usuario puede acceder o no a una vista específica, de forma análoga a las guardas del backend pero con un protagonismo mayor en este proyecto, ya que el control de acceso en el frontend es la primera barrera visible para el usuario (los motivos se desarrollan en la sección~\ref{sec:router_control_acceso}). El directorio \texttt{components/} agrupa componentes reutilizables, organizados en subdirectorios que encapsulan piezas genéricas de interfaz que pueden utilizarse en distintas vistas (por ejemplo, componentes base, formularios y diálogos).

Otros directorios cumplen funciones más acotadas pero igualmente importantes. El directorio \texttt{assets/} contiene recursos estáticos tales como imágenes, iconos y fuentes; \texttt{plugins/} almacena la configuración e inicialización de bibliotecas externas y del propio ecosistema de Vue, como PrimeVue, Pinia o la integración con el router; y \texttt{views/} reúne las vistas principales de la aplicación, cada una asociada a una ruta concreta e integrada con los componentes, las stores y los servicios de la carpeta \texttt{api/}. Finalmente, \texttt{App.vue} y \texttt{main.ts} conforman el punto de entrada de la aplicación: \texttt{main.ts} inicializa Vue, registra los plugins (Pinia, router, PrimeVue, etc.) y monta el componente raíz \texttt{App.vue}, el cual define la estructura general de la interfaz, incluyendo el \emph{layout} y elementos como la barra de navegación.

Toda la lógica compartida se encapsula en \emph{composables} definidos con la Composition API, lo que facilita su reutilización en distintos componentes. Por ejemplo, funciones de validación de formularios, formatos de fecha o interacciones con la API se extraen a composables en lugar de replicarse en cada vista. Además, las rutas se organizan de forma jerárquica, siguiendo el mismo criterio descrito para el backend en la sección~\ref{sec:estructura_jerarquica}, y las guardas de Vue Router emplean la información disponible en las stores para garantizar que sólo los usuarios autorizados accedan a determinadas vistas, replicando así las restricciones definidas en la API. Por último, Vite facilita la configuración y automatiza la importación de componentes y dependencias: su integración con TypeScript y Vue reduce la necesidad de ajustes manuales, mientras que su servidor de desarrollo agiliza la iteración sobre los componentes y la comprobación inmediata de cambios durante el desarrollo.

\subsection{Manejo de estado con Pinia}\label{sec:manejo_estado_pinia}

Las \emph{stores} cobran sentido precisamente porque la API REST del backend es \emph{stateless}: cada petición se procesa de forma independiente, sin que el servidor conserve información de la sesión entre solicitudes.~\cite{gupta_statelessness_rest_2023} Esto simplifica el backend y mantiene la API siempre consistente, pero obliga a que el cliente gestione por su cuenta cualquier información que deba “persistir” durante la navegación (usuario autenticado, token, roles, etc.). Ahí es donde entra \textbf{Pinia} como capa de estado en el frontend.

En términos prácticos, una store actúa como un pequeño almacén de datos en memoria (y, si se desea, persistible en algún tipo de caché del lado del cliente) asociado a un dominio concreto de la aplicación. En este proyecto se definieron, entre otras, las stores \texttt{auth} y \texttt{user}. Ambas se articulan alrededor de la noción de \texttt{currentUser}, pero con responsabilidades diferenciadas:

\begin{itemize}
	\item La store de \texttt{auth} gestiona el \textbf{token de autenticación} y el estado global de sesión. Desde ella se pueden:
	\begin{itemize}
		\item Almacenar y limpiar el token cuando el usuario inicia o cierra sesión.
		\item Regenerar el token cuando el flujo de autenticación lo requiera.
		\item Exponer banderas booleanas como \texttt{isAuthenticated}, calculadas a partir de la presencia (y vigencia) del token en la store.
	\end{itemize}
	Además, es posible incorporar un “reloj interno” o mecanismo de caducidad que, conociendo la duración del token, marque automáticamente \texttt{isAuthenticated = false} cuando éste expire. Este enfoque evita realizar consultas constantes al backend para saber si el token sigue siendo válido: el frontend asume la expiración y, en el momento de detectar que el token deja de ser válido, desencadena los flujos de cierre de sesión, limpieza del token y bloqueo de acceso mediante el router.
	
	\item La store de \texttt{user} mantiene la \textbf{información del usuario autenticado} en un formato rápidamente accesible para las vistas y componentes. Aquí se almacenan datos como el identificador del usuario, su nombre y, especialmente, su lista de roles. Sobre esa lista se implementan utilidades como \texttt{hasAnyRole} o \texttt{rolesList}, que permiten:
	\begin{itemize}
		\item Comprobar si el usuario actual posee uno o más roles específicos.
		\item Derivar banderas como \texttt{isAdmin} en función de la presencia del rol de administrador.
		\item Exponer información de rol de forma reactiva a las vistas, para mostrar u ocultar secciones de la interfaz según los privilegios como se explica en la sección \ref{sec:router_control_acceso}
	\end{itemize}
\end{itemize}

Así, las stores se convierten en las fuentes del estado de autenticación y de los atributos del usuario dentro del frontend, reduciendo el número de peticiones extremo a extremo necesarias (por ejemplo, no es preciso consultar al backend en cada navegación para saber si el usuario es administrador) y aprovecha las ventajas de la memoria en caché del cliente para mantener una experiencia de uso fluida.

\subsection{Enrutamiento y control de acceso}\label{sec:router_control_acceso}

Vue Router es el módulo oficial de enrutamiento de Vue que permite administrar tanto la configuración de rutas como el control de acceso a ellas. En este proyecto se utiliza para definir rutas protegidas y rutas reservadas a determinados roles, más que rutas públicas abiertas a cualquier usuario, ya que el sistema está pensado para funcionar dentro del Departamento de Computación como servicio de inventario y préstamos entre la administración del laboratorio y la comunidad universitaria (estudiantes, profesores y funcionarios). La definición de rutas se realiza a través de la interfaz \texttt{RouteRecordRaw} y de un arreglo \texttt{routes}, donde cada entrada describe el camino de acceso (\texttt{path}), el nombre de la ruta y la vista asociada (\texttt{component}), enlazando directamente con las vistas descritas en la sección~\ref{sec:vistas_componentes}.

Cada ruta definida puede incluir un campo \texttt{meta}, que funciona como un espacio de configuración adicional para el control de acceso. En dicho campo se especifican, por ejemplo, propiedades como \texttt{requiresAuth} (para indicar que la ruta sólo puede visitarse si el usuario está autenticado). De este modo, la ruta raíz \texttt{/} puede declararse con \texttt{meta.requiresAuth = true} para obligar a que el usuario inicie sesión antes de acceder al panel principal, mientras que una ruta como \texttt{/admin} podría incorporar \texttt{meta.requiresAdmin = true}. Junto con \texttt{meta}, cada registro de ruta define el atributo \texttt{component}, que apunta al componente de vista correspondiente. Por ejemplo, la ruta \texttt{/users} se asocia al componente \texttt{UserList}, importado desde la vista \texttt{UserList.vue}; de esta manera, cuando el usuario navega a \texttt{/users}, Vue Router renderiza el componente \texttt{UserList} como contenido principal de la página.

El uso del campo \texttt{meta} se hace efectivo a través de una guard global de navegación, típicamente implementada mediante \texttt{router.beforeEach}. En esta función se consultan las stores de Pinia encargadas de la autenticación y del usuario actual (por ejemplo, \texttt{authStore} y \texttt{userStore}), verificando si \texttt{isAuthenticated} es verdadero cuando la ruta requiere autenticación, y si el usuario posee los roles necesarios cuando la ruta está marcada como \texttt{requiresAdmin}. Si alguna de estas condiciones no se cumple, la guard deniega el acceso y redirige de forma explícita a una ruta alternativa: por ejemplo, hacia \texttt{/login} cuando el usuario no está autenticado o su token ha caducado, o hacia la ruta raíz \texttt{/} cuando un usuario autenticado intenta acceder a una sección exclusiva de administradores. Estas redirecciones pueden complementarse con mensajes de retroalimentación mostrados mediante componentes de notificación (\texttt{Toast}), de modo que el usuario reciba inmediatamente una explicación clara del motivo por el cual se le ha bloqueado el acceso., así, la combinación del conjunto: \texttt{meta}, \texttt{beforeEach} y las stores de Pinia, permiten implementar un control de acceso ordenado, declarativo y fácil de mantener en el frontend, es debido a ello que se hace mayor énfasis y se mantiene una estructura de control de acceso más detallado en este área del sistema, y no en el backend.


\subsection{Comunicación con el backend}\label{sec:comunicacion_apifetch}

La comunicación con el backend se fundamenta en los modelos de datos expuestos mediante los DTOs definidos en Litestar (ver sección~\ref{sec:modelado_datos}). El objetivo es mantener consistencia y coherencia entre las estructuras utilizadas en el servidor y las que se manejan en el cliente. Para ello, en el frontend se definen \texttt{interface} de TypeScript que reflejan la forma de los datos devueltos por la API: por ejemplo, \texttt{interface Token} para la respuesta de autenticación, \texttt{interface Role} para la gestión de roles, \texttt{interface User} para la representación completa de un usuario, o \texttt{interface NewUserPayload} para el cuerpo de las peticiones de creación de usuarios. Estas interfaces actúan como “contratos” tipados que se corresponden con los modelos de SQLAlchemy y los DTOs del backend, permitiendo detectar incongruencias en tiempo de compilación y clarificar el rol de cada tipo dentro del frontend.

Para realizar las peticiones HTTP se construyó un pequeño envoltorio sobre \texttt{fetch}, al que se le pasa un objeto de configuración basado en \texttt{RequestInit}. Esta configuración se tipa mediante una extensión propia, \texttt{RequestInitWithJson}, que simplifica el envío y recepción de datos en formato JSON. Entre otras cosas, este envoltorio se encarga de serializar automáticamente el cuerpo de la petición, establecer las cabeceras adecuadas y aplicar una transformación de nombres de campos desde el formato \texttt{snake\_case} utilizado en el backend (por ejemplo, \texttt{user\_name}) al formato \texttt{camelCase} habitual en TypeScript (por ejemplo, \texttt{userName}).

El resultado de \texttt{fetch} se procesa en una función auxiliar, comúnmente denominada \texttt{APIFetch}, que devuelve directamente la respuesta ya parseada como JSON (o lanza una excepción en caso de error HTTP). Esta función se reutiliza en los archivos de la carpeta \texttt{api},\texttt{auth.ts}, \texttt{users.ts} y \texttt{roles.ts}, donde se declaran funciones específicas para cada endpoint (por ejemplo, \texttt{login}, \texttt{getUsers}, \texttt{createUser}, \texttt{getRoles}, etc.). Cada una de estas funciones construye la URL correspondiente, define el método HTTP y el cuerpo de la petición cuando es necesario, y delega en \texttt{APIFetch} la ejecución efectiva de la llamada al backend. 

\subsection{Componentes y diseño de la interfaz}\label{sec:vistas_componentes}

Las vistas del sistema se implementan mediante archivos \texttt{.vue} que integran de forma concreta todo lo descrito en las secciones~\ref{sec:manejo_estado_pinia}, \ref{sec:router_control_acceso} y \ref{sec:comunicacion_apifetch}. Cada vista combina la lógica de la Composition API, el acceso al estado global a través de las stores de Pinia, la navegación controlada por Vue Router y el consumo tipado de la API mediante los servicios definidos en \texttt{api/}. De este modo, los archivos \texttt{.vue} funcionan como la capa donde se orquestan el flujo de datos y las interacciones del usuario, apoyándose en componentes reutilizables para mantener el código modular y fácil de mantener.

En el directorio \texttt{components/base/} se encuentran los componentes de uso transversal. Un ejemplo clave es \texttt{BaseHeader.vue}, que define la barra de navegación principal (\texttt{<nav>}) del sistema. Este componente se encarga de organizar los enlaces de navegación, utilizando el router para dirigir a los usuarios hacia las distintas vistas según su estado de autenticación y sus roles. A partir de la información expuesta por las stores (por ejemplo, \texttt{isAuthenticated} o \texttt{isAdmin}), \texttt{BaseHeader.vue} muestra u oculta opciones del menú, de modo que un usuario estándar sólo vea las secciones que tiene permitidas, mientras que un administrador accede también a rutas avanzadas, como las dedicadas a la gestión de usuarios o roles.

En el directorio \texttt{components/dialogs/} se incluyen componentes modales que muestran información detallada o requieren confirmación del usuario. Un ejemplo es \texttt{UserDetailsDialog.vue}, que despliega todos los campos relevantes de la interfaz \texttt{User}, ya sea para el usuario autenticado o para un usuario seleccionado en una lista. Este componente se reutiliza en distintas vistas y se alimenta de los datos proporcionados por las stores o por las respuestas de la API, permitiendo inspeccionar el perfil completo sin abandonar la vista principal desde la que se invoca el diálogo.

En todas las vistas y componentes se hace uso intensivo de los elementos proporcionados por PrimeVue, tales como \texttt{Button}, \texttt{Card}, \texttt{Checkbox}, \texttt{Chip}, \texttt{FloatLabel}, \texttt{MultiSelect}, entre otros. Estos componentes se utilizan de forma sistemática en las vistas ubicadas en \texttt{src/views/}, por ejemplo \texttt{HomeView}, \texttt{UserList}, \texttt{UserLogin}, etc., para construir formularios, tablas, tarjetas de información y diálogos. La personalización visual de estos elementos se realiza mediante clases utilitarias de TailwindCSS, lo que permite mantener una estética coherente en todo el sistema sin renunciar a la flexibilidad en el diseño.

\section{Resultados y evidencias}\label{sec:hito3_resultados_evidencias}

En primera instancia, para análisis, inserciones, ediciones y eliminaciones de resultados directamente desde la base de datos, se utiliza \texttt{pgcli} como herramienta de línea de comandos que trabaja con PostgreSQL que es similar al \texttt{psql} que proporciona él mismo, que posee modificaciones de UX mejorada. Generalmente a la base de datos se conecta siguiendo el siguiente formato:

\begin{lstlisting}[language=bash, numbers=none]
	$ pgcli -h localhost -U [mi_usuario] -d [database]
\end{lstlisting}

En este caso concreto, \texttt{database=labdic\_inventory}, y se pueden realizar operaciones SQL para testing de la base de datos, como por ejemplo:

\begin{lstlisting}[language=SQL, numbers=none]
	-- 1) Insertar un nuevo usuario en la tabla users
	INSERT INTO users (username, email, password)
	VALUES ('ariel', 'ariel@example.com', 'mi_contrasena_segura');
\end{lstlisting}


\begin{lstlisting}[language=SQL, numbers=none]
	-- 2) Modificar un rol existente en la tabla roles
	UPDATE roles
	SET name        = 'lab_admin',
	description = 'Administrador del laboratorio y del inventario'
	WHERE id = 2;
\end{lstlisting}

\begin{lstlisting}[language=SQL, numbers=none]
	-- 3) Eliminar una relacion usuario-rol en la tabla user_roles
	DELETE FROM user_roles
	WHERE user_id = 5
	AND role_id = 2;
\end{lstlisting}

Seguido de esta corroboración, se accede a los resultados desde el backend utilizando la implementación automática de OpenAPI que ofrece Litestar, con el fin de realizar pruebas directas sobre la base de datos a través de las APIs creadas y descritas en la sección~\ref{sec:modulo_usuario_backend}.

Al inicializar el entorno virtual del backend, desde el directorio correspondiente, mediante terminal con:

\begin{lstlisting}[language=bash, numbers=none]
	$ source ./.venv/bin/activate
	$ uv run litestar run --reload
\end{lstlisting}

la API de Litestar queda disponible en la ruta \texttt{localhost:8000/schema/}, donde en este caso se utiliza \texttt{Scalar} como interfaz gráfica para explorar e invocar los endpoints (de forma equivalente a \texttt{Swagger} u otras herramientas similares). Esta interfaz permite autenticar un usuario, enviar peticiones HTTP a cada recurso y observar en tiempo real tanto las cabeceras como los cuerpos de respuesta.

En la Figura~\ref{fig:scalar_overview} se muestra la vista general de la \emph{Labdic Inventory API} en Scalar. En la columna izquierda se listan los grupos de endpoints organizados por recurso (\texttt{users}, \texttt{roles} y \texttt{auth}), con las operaciones CRUD correspondientes para cada uno de ellos (\texttt{ListUsers}, \texttt{CreateUser}, \texttt{GetUser}, etc.). En la parte derecha se encuentra el panel de autenticación, donde se configura el esquema \emph{BearerToken} asociado a OAuth2. Allí se indica la URL del endpoint de login (\texttt{/labdic\_inventory/auth/login}) y se ingresan las credenciales de prueba (nombre de usuario y contraseña) para obtener un token de acceso. Una vez presionado el botón \texttt{Authorize}, Scalar almacena el token y lo adjunta automáticamente a las solicitudes que requieren autenticación.
